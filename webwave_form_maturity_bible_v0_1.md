# Preface: What This Document Is

This document is the maturity bible for WebWave. It is not a minimum viable product plan, not a pitch deck, and not a near-term execution checklist. It is the stable reference for the thing in its richest mature form: what the product wants to become if the core thesis works, the right constraints are honored, and the project is allowed to grow into its highest shape.

The original design journal established the seed: a browser-native game where every URL can become a playable level. The mature synthesis sharpened that into a larger product thesis: WebWave is best understood as a playable atlas of the web, not merely a novelty generator. It is a game, a ritual, a map, an archive, a social ghost layer, and eventually a creator platform for playable lenses.

The center of gravity is this:

> WebWave makes the public web playable. The URL is the level, the share link, the scoreboard, the memory, and the cultural object. AI is the lens that reads the page. Designers and creators build the verbs. Players leave ghosts. The page remembers.

This bible should guide future decisions. If a proposed feature strengthens the playable link, the Level Compiler, the Atlas, the archive, the ghost layer, the daily ritual, or the lens creator ecosystem, it probably belongs. If it turns the product into a generic game engine, a surveillance extension, an AI-content detector, or an endless pile of uncurated modes, it probably does not.

# 1. North Star

## 1.1 One-sentence vision

WebWave is a browser-native game and creative platform where public web pages become playable, scoreable, shareable, archivable levels, and where players and creators collectively build a living atlas of the internet through play.

## 1.2 The mature product in one paragraph

At maturity, WebWave is a daily game, a browser extension, a public URL atlas, a persistent ghost layer, a playable page archive, and a creator marketplace of lenses. A player can visit the WebWave site and play the daily page with everyone else, install the extension and summon the game on public pages, compete on per-URL leaderboards, race against ghosts, browse historical versions of pages, discover curated page expeditions, and eventually apply or create alternate lenses that interpret different kinds of web pages through different playable verbs. The public web becomes terrain. URLs become arenas. Page changes become history. Player traces become memory. Creator lenses become ways of reading the web through action.

## 1.3 The product category

WebWave should not be positioned as only one thing. Its durable category is a hybrid:

- Browser-native skill game.
- Daily challenge ritual.
- Procedural level system.
- Social asynchronous competition layer.
- Playable web archive.
- UGC lens platform.
- Cultural map of the web expressed through gameplay.

The dangerous simplification is: "AI turns websites into games." That is catchy, but shallow. The better formulation is:

> WebWave is the playable link.

# 2. The Core Thesis

## 2.1 The web is already structured like a game world

Every page contains signals: headings, paragraphs, links, forms, ads, images, whitespace, color, rhythm, hierarchy, repetition, density, friction, persuasion, motion, and intent. WebWave treats those signals as playable material. It does not need to invent a world from nothing. The web is already full of structure. The game compiles that structure into action.

A page is not only a visual document. It is also a behavioral object. It asks for attention, clicks, scrolling, trust, scanning, purchasing, reading, judging, comparing, navigating, and remembering. WebWave turns those implicit demands into explicit gameplay.

## 2.2 AI is the lens, not the author

The original journal's most important architectural decision should remain permanent: WebWave should not rely on AI to generate arbitrary gameplay from scratch. The AI should read, classify, summarize, and help parameterize. The verbs, scoring, physics, constraints, and validators should remain human-authored and deterministic.

This matters because game feel requires fairness, repeatability, readability, and polish. Generative world models are improving, but even advanced interactive models still face limitations around action control, consistent long-duration interaction, multiplayer simulation, text rendering, and exact representation. The mature WebWave should therefore use AI where it is strongest: interpretation, classification, curation assistance, flavor generation, and tooling. It should not make AI the uncontrolled game engine.

The rule:

> AI reads. Designers make. The compiler binds the two.

## 2.3 The commentary is structural, not accusatory

WebWave is about the modern web, including AI-mediated content, advertising density, SEO patterns, commercial pressure, and the difference between human texture and synthetic regularity. But it should never become a public accusation engine. It should not label pages as "AI slop," "propaganda," "fake," "bad," or "human." It should translate page qualities into feel.

A cluttered page feels cluttered. A clean page feels clean. A repetitive page feels smooth and synthetic. A human-made personal page feels irregular, warm, and surprising. An ad-heavy page physically pressures the player. A bureaucratic page feels rigid. A recipe page with intrusive monetization becomes a comic gauntlet. The player feels the page through hands and ears, not through a scolding label.

## 2.4 The promise is not that every page is equally fun

A core mature-state reframing:

> WebWave should not promise that every URL is equally fun. It should promise that every public page has a playable signature, and that the community discovers, ranks, curates, haunts, archives, and competes on the pages that become great levels.

This is crucial. Infinite content alone does not create retention. Infinite terrain needs trails, landmarks, rituals, rankings, seasons, ghosts, memories, and curators.

The best WebWave is not a machine that magically makes every URL great. It is a system where:

- Every public page can be read.
- Many pages can produce playable signatures.
- Some pages become excellent levels.
- A few become legendary.
- The community finds them.
- The Atlas remembers them.

# 3. What WebWave Is and Is Not

## 3.1 WebWave is

WebWave is a game of attention, structure, rhythm, and interpretation. It turns public pages into compact action experiences. It lets players compete on pages, discover page territories, replay archived versions, and leave traces for others. It eventually lets creators design lenses that apply new playable interpretations to classes of pages.

It is a way of saying: the web is a world, and links are levels.

## 3.2 WebWave is not

WebWave is not a generic game maker. A mature creator system should not become "Roblox for websites." WebWave's creator layer is about lenses that interpret the web, not arbitrary games that happen to be launched from URLs.

WebWave is not raw AI-generated gameplay. The system should not ask a model to invent a full game loop for each URL. That would be hard to balance, hard to score, hard to moderate, hard to replay, and hard to trust.

WebWave is not an AI detector. Even if the system uses signals related to synthetic regularity, it should not publicly assert that a page is AI-generated or human-generated.

WebWave is not a surveillance extension. The mature extension should be user-invoked and privacy-forward, preferably relying on temporary active-tab access rather than broad always-on page access wherever possible.

WebWave is not a scraper that reproduces web content. It should extract abstract features, stylize page signatures, and avoid copying text or screenshots into the public game layer.

WebWave is not a social network pasted on top of pages. Its social presence should feel like haunting, not notification. Ghosts, score curves, traces, and page memory are better than chat spam.

# 4. The Mature Product Pillars

## 4.1 Pillar One: The Canonical Game

Every public playable URL should have an official canonical WebWave interpretation. This is the default mode, the main leaderboard, the daily challenge version, and the stable competitive surface.

The canonical layer provides clarity. It says: when someone says "I played this page," everyone understands the default run. Community lenses can add alternate interpretations, but they should not erase the main game.

Canonical WebWave must be:

- Learnable in seconds.
- Skill-based rather than random.
- Fair across devices and viewport classes.
- Stable across page versions.
- Replayable enough for competition.
- Sensory enough to feel tied to the source page.
- Safe enough for public leaderboards.

## 4.2 Pillar Two: The Level Compiler

The Level Compiler is the central technical and creative instrument. It reads page signals and turns them into playable scripts.

The compiler is not only a feature vector mapper. It should eventually understand page anatomy, create multi-act structures, select verbs, tune difficulty, define scoring opportunities, produce visual and audio mappings, generate ghosts and bot benchmarks, and validate solvability.

The compiler is the heart of the product. If the compiler is shallow, WebWave becomes a gimmick. If the compiler is rich, WebWave becomes a new media form.

## 4.3 Pillar Three: The Atlas

The Atlas is the browsable world map of played pages, domains, lenses, histories, and player discoveries. It is not merely analytics. It is the overworld.

The Atlas should show:

- Pages as territories.
- Domains as regions.
- Daily pages as monuments.
- Great URL/lens pairings as landmarks.
- Archived versions as fossils.
- Player traces as paths.
- Hard pages as mountains.
- Clean pages as gardens.
- Ad-heavy pages as storms.
- Weird personal pages as hidden shrines.

The Atlas makes the game durable because it turns isolated runs into accumulated world-building.

## 4.4 Pillar Four: The Ghost Layer

WebWave's social presence should feel like evidence that others have been here. The player should sense the web is haunted by previous runs.

Ghosts should show decisions rather than avatars. The system should show where great players nearly died, what they prioritized, how their score curve developed, where friends panicked, where the AI benchmark took a strange route, and where the crowd collectively struggled.

This creates asynchronous competition and memory without turning pages into chat rooms.

## 4.5 Pillar Five: The Archive

Web pages change. WebWave should preserve playable versions when those changes matter. A URL's history should be browsable as a timeline of playable states.

The mature archive is more than technical versioning. It is one of the project's most poetic features. It allows players to experience:

- A company homepage before and after a redesign.
- A news page across a breaking event.
- A product launch page across hype cycles.
- A personal blog over years.
- A government page before and after a policy shift.
- A page that became viral and then decayed.

The archive turns page change into playable time.

## 4.6 Pillar Six: The Daily Ritual

The daily page concentrates attention. It solves cold start, creates a shared reference, trains new players, gives the game a social heartbeat, and creates recurring cultural moments.

The daily page should be snapshot-stable, curated, safe, and varied. It is the Wordle-like ritual: one page everyone plays, one leaderboard everyone understands, one share card that makes sense.

## 4.7 Pillar Seven: The Summon Extension

The extension unlocks "play any public URL," but it should feel like summoning the game, not like being watched. The user chooses when to activate WebWave. The page becomes playable because the player called the game into being.

The preferred mental model:

> Press the button. This page becomes a level.

Not:

> Install this thing and it watches every page.

## 4.8 Pillar Eight: Lens Studio and the Lens Market

The mature creator system is a marketplace of playable lenses for the web. Creators do not simply build isolated levels. They create ways of interpreting pages through play.

A lens is a reusable interpretation system. It defines signals, verbs, mappings, act structure, scoring, visual treatment, audio treatment, accessibility constraints, validation rules, and recommended URL families.

This is not an early feature. It is a mature-state platform layer built on the same internal tools the WebWave team uses to design and tune official templates.

# 5. The Player Experience at Maturity

## 5.1 First-time player

A new player arrives through a share card or the daily page. They do not need to understand AI, extraction, compilers, or page features. They see a URL they recognize or are curious about. They press play. In seconds, they are moving, tapping, dodging, collecting, slicing, sorting, or timing.

The first-run experience should answer four questions without explanation:

1. What am I doing?
2. Why does this feel like this page?
3. How did I do?
4. Why should I send this to someone?

The first play must feel immediate and bodily. The explanation can come later.

## 5.2 Daily player

The daily player treats WebWave as a ritual. They check today's page, compare their percentile, replay once or twice, share a card, maybe inspect ghosts, then leave. The daily player does not need to be a deep explorer. They are the heartbeat of the public surface.

The daily product must support:

- Fast load.
- Snapshot fairness.
- Strong share card.
- Daily streak.
- Clear leaderboard percentile.
- Top ghost comparison.
- Short session loops.
- Mobile and desktop play.

## 5.3 Competitor

The competitor cares about mastery, leaderboards, ghosts, normalized scores, official runs, and tournament pages. They need stable scoring, anti-cheat, repeatable versions, bot benchmarks, and deep stats.

The competitor wants to know:

- Which pages are hard?
- Which templates am I best at?
- Where did the top player gain score?
- Can I beat the AI benchmark?
- Can I master this domain or page category?
- Can I hold a record on a famous URL?

The competitor creates prestige.

## 5.4 Explorer

The explorer treats the web as terrain. They install the extension, play strange pages, discover great levels, curate playlists, and fill out their personal map.

The explorer wants:

- Pioneer badges.
- Domain sweeps.
- Hidden page discoveries.
- Great URL/lens pairings.
- Personal Web Map growth.
- Atlas visibility for discoveries.
- Collections and trails.

The explorer creates breadth.

## 5.5 Archivist

The archivist cares about change over time. They replay old versions, compare page redesigns, find historical shifts, and create expeditions through page timelines.

The archivist wants:

- Version history browser.
- Page portrait timelines.
- Difference lenses.
- Frozen old leaderboards.
- Archived ghosts.
- Historical collections.
- Cultural context around page changes.

The archivist gives the project memory.

## 5.6 Creator

The creator uses Lens Studio to invent new ways to play the web. They may create broadly useful lenses, single-URL masterpieces, page-category jokes, critical interpretations, accessibility-first modes, or high-skill competitive variants.

The creator wants:

- A safe editor.
- Fast preview.
- URL test benches.
- Signal mapping.
- Act timelines.
- Validation.
- Publication tiers.
- Remix lineage.
- Featured opportunities.
- Creator identity.

The creator gives the project renewal.

## 5.7 Domain owner or cultural partner

A domain owner, publisher, museum, creator, school, or brand may want to understand how their public pages feel through WebWave. They might sponsor a page, claim a domain, host a collection, or use aggregate playability analytics.

This persona should never become the main player at the expense of consumer love. But in the mature business, domain portals and cultural partnerships can become meaningful revenue and legitimacy.

# 6. The Level Compiler

## 6.1 Why the compiler is the heart

The Level Compiler is the difference between a gimmick and a durable product. A shallow version says: "high text density means more obstacles." A mature version says: "this page has a long formal opening, a dense persuasive middle, repeated CTAs, an image cluster, and a noisy footer; create a four-act level where the first act teaches the rhythm, the second applies pressure, the third branches, and the final act tests attention under clutter."

The compiler must make pages feel authored without being handcrafted individually.

## 6.2 Compiler pipeline

The mature pipeline should look like this:

1. User invokes WebWave or plays a curated snapshot.
2. Page is checked for safety, privacy, eligibility, and canonicalization.
3. Client extracts deterministic signals from the rendered page.
4. Volatile and unsafe elements are stripped or downweighted.
5. Signals are normalized by viewport class, domain patterns, and page type.
6. Optional AI lens classifies mood, intent, content type, semantic structure, and page-role signals.
7. Compiler builds a page anatomy model.
8. Compiler selects canonical template or hybrid verb plan.
9. Compiler creates a multi-act level script.
10. Sensory renderer maps color, rhythm, noise, texture, and audio.
11. Scoring contract is generated from official rules.
12. Bot validator tests solvability, difficulty, exploitability, and fun proxies.
13. Cache stores feature signature, parameterization, version key, and validation report.
14. Game runs client-side.
15. Score, trace, and ghost data are submitted asynchronously.

## 6.3 Page anatomy model

A mature compiler should identify sections such as:

- Hero or header zone.
- Navigation cluster.
- Main body.
- Paragraph rhythm segments.
- Image/media clusters.
- Product grids.
- Forms and inputs.
- CTA clusters.
- Ad and sponsorship zones.
- Sidebar modules.
- Comments and social embeds.
- Footer and legal zones.
- Infinite-scroll continuation.
- Popups, banners, overlays, and sticky elements.

Each section can become an act, transition, obstacle family, rhythm segment, reward pocket, rest zone, or boss event.

Commercial pressure signals:

## 6.5 From feature vector to level script

The mature output should be a level script, not just parameters. A level script contains:

- Page identity.
- Version identity.
- Template or lens identity.
- Act sequence.
- Event timeline.
- Spawn rules.
- Object families.
- Movement constraints.
- Difficulty curve.
- Scoring contract.
- Visual palette.
- Audio palette.
- Ghost trace schema.
- Accessibility transformations.
- Validation results.

Example simplified schema:

```json
{
  "canonical_url": "https://example.com/article",
  "page_version_id": "v17",
  "lens_id": "official.rhythm_collect.hybrid",
  "viewport_class": "desktop.medium",
  "acts": [
    {"name": "hero_scan", "duration": 12, "verbs": ["collect"], "intensity": 0.25},
    {"name": "body_rhythm", "duration": 45, "verbs": ["rhythm"], "intensity": 0.55},
    {"name": "ad_pressure", "duration": 20, "verbs": ["surge", "defend"], "intensity": 0.80},
    {"name": "footer_release", "duration": 10, "verbs": ["collect"], "intensity": 0.35}
  ],
  "scoring": {"accuracy": 0.4, "efficiency": 0.2, "endurance": 0.2, "combo": 0.2},
  "validation": {"solvable": true, "fps_estimate": 60, "difficulty_band": "medium"}
}
```

## 6.6 Fun validation

The validator cannot know fun perfectly. It can catch many failures:

- Impossible density.
- Unreadable object overlap.
- Unfair spawn timing.
- Degenerate score loops.
- Device-specific unfairness.
- Too little interaction.
- Too much randomness.
- Excessive motion for accessibility settings.
- No meaningful connection to page signals.
- Bad score distribution.
- Bot exploit routes.

The mature validator should use deterministic checks, scripted bots, simulation runs, player outcome data, ratings, and retention signals.

## 6.7 Fun-yield as a central metric

A key mature metric should be fun-yield:

> The percentage of pages in a given URL class that produce levels players rate as fun, fitting, fair, and replayable.

The goal is not universal excellence. The goal is to know which page classes, lenses, and compiler settings produce high fun-yield, then route players and creators toward those zones.

# 7. Official Verb Families

## 7.1 Design rule for every verb

Every official verb must be fun without the web layer. A template that is not fun as a standalone microgame will not be saved by page extraction. WebWave should treat each verb like a compact arcade game with its own skill ceiling.

Each verb should be:

- Learnable in under five seconds.
- Playable on touch, mouse, keyboard, and relaxed modes.
- Visually readable at high speed.
- Scoreable with clear skill dimensions.
- Parameterizable across density, rhythm, difficulty, palette, and mood.
- Compatible with ghost traces.
- Capable of forming hybrid acts with other verbs.

## 7.2 SURGE

SURGE is the dodge and movement verb. The player moves through a scrolling pressure field. Obstacles represent density, clutter, commercial pressure, or structural friction.

Best page sources:

- News sites.
- E-commerce pages.
- Ad-heavy pages.
- Corporate pages with many modules.
- Busy landing pages.

Mature enrichments:

- Different obstacle species for ads, CTAs, sticky headers, repeating modules, and chaotic layout.
- Calm lanes generated by whitespace.
- Ghost danger markers from top players.
- Burst events from commercial pressure.
- Page-section acts, rather than uniform scrolling.

## 7.3 COLLECT

COLLECT is the value-prioritization verb. The player gathers meaningful objects before they fade, learning to distinguish signal from filler.

Best page sources:

- Wikipedia and reference pages.
- Essays and blogs.
- Recipe pages.
- Documentation pages.
- Mixed-media pages.

Mature enrichments:

- Item value based on semantic role, not just element type.
- Filler objects that tempt but lower efficiency.
- Chain bonuses for prioritizing high-signal content.
- Collection routes that reveal page structure.
- Accessibility-friendly relaxed modes.

## 7.4 SLICE

SLICE is the precision gesture verb. Elements fly across the screen, and the player slices according to timing, direction, type, or combo state.

Best page sources:

- Image-heavy pages.
- Portfolios.
- Galleries.
- Visual essays.
- Product pages with many cards.

Mature enrichments:

- Slice directions mapped to media clusters and layout orientation.
- Combo windows based on visual rhythm.
- Misdirection objects from repeated promotional modules.
- Charging, parrying, and perfect-cut skill layers.

## 7.5 RHYTHM

RHYTHM is the timing verb. Sentence length, paragraph cadence, headings, lists, quotes, and whitespace become beats, rests, holds, and syncopations.

Best page sources:

- Long articles.
- Essays.
- Documentation.
- Academic pages.
- Speeches and transcripts.

Mature enrichments:

- Paragraphs as phrases.
- Headings as downbeats.
- Quotes as sustained notes.
- Footnotes/citations as grace notes.
- Dense or repetitive prose as texture.
- Strong accessibility options for tempo and input.

## 7.6 BALANCE

BALANCE is the sorting and judgment verb. Page elements must be sorted into categories under pressure.

Best page sources:

- Product listings.
- Comparison pages.
- Directories.
- Forms.
- Categorical reference pages.

Mature enrichments:

- Category ambiguity as difficulty.
- Multi-bin sorting for expert mode.
- Semantic similarity as tension.
- Wrong-sort ghost outlines.
- Modes where the player learns the taxonomy through play.

## 7.7 TRACE

TRACE is the navigation verb. The page's link graph, headings, anchors, and internal structure become a maze or route-planning game.

Best page sources:

- Wikis.
- Documentation.
- Government sites.
- Directories.
- Large institutional pages.

Mature enrichments:

- Internal links become corridors.
- External links become portals.
- Breadcrumbs become safe paths.
- Link farms become traps.
- The player's route becomes a ghost path on the Atlas.

## 7.8 STACK

STACK is the organization verb. Page elements become pieces that must be arranged into stable structures.

Best page sources:

- Layout-heavy pages.
- Product cards.
- Design portfolios.
- Dashboards.
- Listicle pages.

Mature enrichments:

- DOM hierarchy determines piece nesting.
- Visual weight affects physics.
- Clean pages produce elegant structures.
- Chaotic pages produce unstable towers.
- Collaborative score goals can emerge in seasonal modes.

## 7.9 DEFEND

DEFEND is the attention-protection verb. Waves of noise, popups, CTAs, autoplay modules, or distractions attempt to invade a protected focus zone.

Best page sources:

- News sites.
- Recipe sites.
- Affiliate pages.
- Marketing pages.
- Social embeds.

Mature enrichments:

- Ad clusters become waves.
- Newsletter boxes become shield breakers.
- Sticky bars become pressure fronts.
- Meaningful content must be protected rather than collected.
- Great for comedic and critical lenses.

## 7.10 RACE

RACE turns scroll-depth and page structure into terrain. The player races through a track derived from the page's layout.

Best page sources:

- Long landing pages.
- Scrollytelling pages.
- Product launches.
- Articles with rich sectioning.

Mature enrichments:

- Hero sections as launch ramps.
- Dense body sections as technical turns.
- Ads as hazards.
- Whitespace as speed lanes.
- Page version changes as new track variants.

## 7.11 ECHO

ECHO is a ghost-focused verb. The player interacts with prior traces, crowd heat, and their own past runs.

Best page sources:

- Popular daily pages.
- Archived versions.
- Tournament pages.
- Pages with rich replay data.

Mature enrichments:

- Crowd paths become moving currents.
- Friend traces become optional guides.
- Top-player decisions become fleeting opportunities.
- The player can race their previous interpretation.

## 7.12 FREEFORM

FREEFORM is expert-mode synthesis. Multiple verbs are active at once or can be switched dynamically. It should be rare, late, and earned.

Best page sources:

- Complex pages with strong sectioning.
- Featured events.
- Creator lenses.
- Boss pages.

Mature enrichments:

- Multi-act hybrid modes.
- Player-selected verb loadouts.
- High skill ceiling.
- Creator experimentation.

# 8. Sensory Translation

## 8.1 Why sensory translation matters

The mechanical layer makes the page playable. The sensory layer makes it believable. Without sensory fidelity, WebWave becomes "same game, different density." With sensory fidelity, players can feel that a corporate page, a personal blog, a recipe page, and a government form are different worlds.

## 8.2 Visual language

Mature visual translation should include:

- Palette derived from page colors, but stylized rather than copied.
- Density mirrored through object count, background texture, and spacing.
- Whitespace represented as calm, rest, open lanes, or reverb-like visual space.
- DOM complexity represented as layering, parallax, occlusion, or jitter.
- Commercial pressure represented as sharpness, compression, warning rhythm, or burst behavior.
- Synthetic regularity represented as smoothness, uniformity, glassiness, or too-perfect repetition.
- Human texture represented as warmth, asymmetry, irregular timing, and handcrafted variation.
- Page type represented through shape grammar, not labels.

## 8.3 Audio language

Audio should make each page feel like it has a sonic identity. The mature audio system should be procedural and parameterized, not composed individually.

Mappings:

- Page energy -> tempo.
- Text rhythm -> beat pattern.
- Whitespace -> rests and reverb space.
- Density -> layer count.
- Commercial pressure -> sharper transients.
- Formality -> steadier tone and stricter grid.
- Personal/creative pages -> warmer, less quantized sounds.
- Synthetic regularity -> smoother timbres and looped motifs.
- Human irregularity -> subtle timing variation.

Minimum mature audio should include responsive sound atoms, ambient layers, combo motifs, page-type timbres, and accessibility controls.

## 8.4 Page portraits

The page portrait is not a screenshot. It is a stylized signature generated from the feature vector and level script. It should become a collectible visual language. Regular players should learn to read it:

- Dense portrait -> intense run.
- Minimal portrait -> clean rhythm.
- Fragmented portrait -> chaotic layout.
- Smooth synthetic portrait -> regularized content texture.
- Warm irregular portrait -> human-feeling page.
- Radial link portrait -> navigation-heavy page.

The portrait appears on share cards, Atlas nodes, version timelines, and lens cards.

# 9. URL Identity, Versioning, and Fairness

## 9.1 URL identity is not enough

A mature leaderboard cannot simply use raw URL as identity. Web pages vary by viewport, locale, logged-in state, A/B test, ads, query parameters, personalization, and time. WebWave needs a canonical key.

A mature key should consider:

- Canonical URL.
- Normalized query parameters.
- Content signature.
- Feature signature.
- Page version.
- Viewport class.
- Locale/language.
- Template or lens version.
- Compiler version.
- Accessibility-adjusted mode.

Example:

```text
canonical_url + page_version_id + lens_id + lens_version + compiler_version + viewport_class + locale
```

## 9.2 Stable public competition

Official leaderboards must be stable and comparable. If page changes materially affect gameplay, a new version should be created. If minor volatility does not change gameplay meaningfully, the leaderboard should continue.

Versioning rule:

- Similar feature signature -> same version.
- Meaningful change but same gameplay -> same version with metadata note.
- Perceptible gameplay change -> new active version.
- Major page transformation -> new version and archive prior one.

## 9.3 Archived versions

Old versions should remain playable. Leaderboards freeze by version. Ghosts remain attached to the version that created them. The URL's page portrait evolves over time.

This creates the playable archive.

# 10. Ghosts, Traces, and Social Presence

## 10.1 Haunting, not notification

The mature social principle:

> The player should feel others have been here, not be interrupted by them.

Avoid intrusive social feeds, chat boxes, alerts, and notification spam. Use traces, ghosts, score curves, and ambient presence.

## 10.2 Trace types

Official trace types:

- Top Ghost: the best public score route.
- Friend Ghost: selected friends or private group runs.
- First Explorer Ghost: the first human to play the page.
- AI Benchmark Ghost: machine route and score target.
- Crowd Echo: aggregate heatmap of many runs.
- Past Self: the player's own best prior run.
- Daily Crowd: transient echo from today's public challenge.

## 10.3 Trace by verb

SURGE traces show near-miss danger markers and route pressure.

COLLECT traces show priority choices and efficient sweep paths.

SLICE traces show timing arcs, perfect cuts, and missed opportunities.

RHYTHM traces show beat accuracy and combo breaks.

BALANCE traces show sort-bin decisions and ambiguity zones.

TRACE traces show path choices and dead ends.

DEFEND traces show where attention broke.

STACK traces show structural stability and collapse points.

## 10.4 Score curve

The universal ghost is the score curve. It works across all verbs. A thin curve shows the player's score growth over time, compared with top ghost, friend ghost, AI benchmark, or past self.

The score curve answers the core competitive question:

> Am I ahead or behind, and where did the run turn?

# 13. Extension, Privacy, and Trust

## 13.1 The summon model

The mature extension should use a summon model. It should not feel always-on. It should process a page only when the user chooses to play or explicitly enables a preview feature.

Preferred flow:

1. User clicks extension, keyboard shortcut, context menu, or WebWave button.
2. Extension receives temporary access to the active tab where possible.
3. It checks safety and eligibility.
4. It extracts abstract features locally.
5. It sends only permitted, non-reconstructive signals or a minimal summary when needed.
6. It loads the cached or compiled game.
7. It clears transient sensitive state after use.

This is both a product trust decision and a technical architecture decision.

## 13.2 Data minimization

WebWave should minimize data at every stage:

- Do not transmit raw page text by default.
- Do not transmit screenshots.
- Do not log browsing history.
- Do not process private/authenticated pages for public play.
- Do not create public records for sensitive pages.
- Hash and normalize URLs carefully.
- Store feature vectors only where necessary.
- Give users visibility into extracted features.
- Provide deletion controls.
- Use domain opt-out and claim mechanisms.

## 13.3 User-viewable extraction

A mature trust feature should be "What WebWave read." Before or after first play, the user can inspect a simple report:

- Page type.
- Signals extracted.
- What was sent to the server.
- What stayed local.
- Whether raw text was excluded.
- Why a page was blocked or private-only.

This turns privacy from a policy into a visible product affordance.

## 13.4 Sensitive page blocklist

Block by default:

- Banking.
- Payment dashboards.
- Email.
- Health portals.
- Insurance portals.
- Government authentication.
- Login/account pages.
- Password managers.
- Cloud storage private files.
- Adult content domains.
- Private social feeds.
- Messaging apps.
- School/student portals.
- Workplace admin systems.

The user should be able to add custom blocklist domains.

## 13.5 Public/private separation

Eligibility tiers:

Public Competitive: safe public pages, full leaderboard, ghosts, Atlas inclusion.

Public Unlisted: safe enough to share directly, not surfaced in discovery.

Private/Friend Mode: no public leaderboard, no Atlas inclusion, limited traces.

Blocked: no gameplay or only a local sandbox with no extracted data leaving the device, depending on legal and safety decisions.

This tiering protects users, domains, and the project.

# 14. Safety, Moderation, and Legal Posture

## 14.1 Safety principle

Public play is a privilege, not a default. A page should earn public competitive status through safety checks, public accessibility, and low privacy risk.

## 14.2 Content safety

Before public leaderboard creation, WebWave should screen for:

- Explicit sexual content.
- Hate and extremist content.
- Graphic violence.
- Illegal services or goods.
- Self-harm encouragement.
- Doxxing or personal data exposure.
- Private/authenticated content.
- High-risk misinformation contexts where surfacing could amplify harm.

Failed pages may be blocked, private-only, or unlisted depending on severity.

## 14.3 Creator safety

Creator lenses add another layer of risk. They should not be allowed to:

- Make explicit factual accusations about pages.
- Reproduce page content beyond allowed abstract signals.
- Target protected groups.
- Encourage harassment of domains or people.
- Create unsafe flashing/motion patterns.
- Use arbitrary external network calls.
- Execute arbitrary code.
- Collect personal data.
- Manipulate official leaderboards.

## 14.4 Domain rights and opt-outs

WebWave should provide:

- Domain owner claim flow.
- Public opt-out for Atlas surfacing.
- Appeal process for public page listings.
- Clear explanation that WebWave uses abstract features and does not reproduce page content.
- Robots.txt respect for server-side pre-warming.
- Different rules for client-invoked user play and server-side crawling.

Legal counsel is required before launch, especially around public pages, derivative visual signatures, sponsored pages, domain claims, and data protection.

## 14.5 Avoid screenshots as core assets

The product should avoid using screenshots of third-party pages in share cards, portraits, leaderboards, and Atlas nodes. Stylized feature-vector portraits are safer, more distinctive, and more aligned with the thesis.

# 15. AI Philosophy and Model Use

## 15.1 AI jobs

AI should do these jobs:

- Classify page type and mood.
- Interpret semantic structure.
- Estimate energy, formality, and synthetic regularity as internal signals.
- Assist daily-page curation.
- Generate share-card highlight text.
- Help creators draft lens graphs.
- Suggest URL test packs.
- Identify possible scoring exploits.
- Power AI benchmark bots or bot personalities when appropriate.
- Help summarize aggregate domain analytics.

AI should not:

- Invent arbitrary game rules per URL.
- Make public accusations about page authorship or quality.
- Be the sole judge of safety.
- Generate non-deterministic scoring logic.
- Run arbitrary creator-submitted code.
- Replace human playtesting of official verbs.

## 15.2 Synthetic regularity, not AI density

The original feature vector included an AI-density estimate. Mature product language should avoid that public framing.

Use internal terms like:

- Synthetic regularity.
- Surface homogeneity.
- Repetition smoothness.
- Texture uniformity.
- Semantic redundancy.

These can affect sensory translation, but should not be shown as claims.

## 15.3 AI benchmark bots

The AI benchmark should become more than a cold-start number. It can become a set of recognizable bot personalities:

- Scanner: safe, efficient baseline.
- Greedy Bot: high value, high risk.
- Rhythm Bot: timing specialist.
- Chaos Bot: strange route, high variance.
- Median Human Bot: learned from aggregate human behavior once enough data exists.

Bots give first plays a target and reinforce the human-vs-machine theme without making the product preachy.

# 16. Lens Studio and the Lens Market

## 16.1 Why creator lenses fit

The creator layer fits because WebWave is fundamentally about interpreting pages through play. If the Level Compiler is the instrument, Lens Studio lets skilled creators compose new interpretations.

The mature creator object should be a Lens, not a generic level.

A Lens is:

> A reusable playable interpretation system that maps page signals into verbs, acts, scoring, visuals, audio, accessibility constraints, and validation rules.

This is a marketplace of ideas and interpretation marketed as games.

## 16.2 What a lens contains

A mature Lens package should include:

- Name.
- Creator.
- Description.
- Best page types.
- Allowed verbs.
- Signal mappings.
- Act timeline rules.
- Difficulty curves.
- Scoring contract.
- Visual style rules.
- Audio style rules.
- Accessibility profile.
- Validation constraints.
- Bot benchmark settings.
- Certified URL list.
- Known failure modes.
- Version history.
- Remix lineage.
- Moderation status.

## 16.3 The editor: URL Bench

The URL Bench is where creators test lenses against one or more pages.

Features:

- Add a specific URL.
- Add a page category pack.
- Add WebWave stress-test packs.
- Compare outputs across pages.
- See feature signatures.
- Inspect page anatomy.
- Mark sections as useful or noisy.
- Save certified URL/lens pairings.

The creator starts from real web material, not a blank canvas.

## 16.4 The editor: Signal Mapper

The Signal Mapper lets creators bind page signals to playable events.

Example mappings:

- Ad density -> obstacle waves.
- Heading frequency -> checkpoints.
- Link clusters -> maze nodes.
- Product cards -> sortable objects.
- Whitespace -> rest zones.
- Long paragraphs -> sustained notes.
- Repetition -> echo enemies.
- CTA buttons -> gates.
- Images -> slice targets.
- Comments -> swarm outro.

The interface should be visual, immediate, and constrained.

## 16.5 The editor: Act Timeline

The Act Timeline lets creators shape a lens into phases.

Example:

- Opening: scan the hero.
- Act 1: collect meaningful objects.
- Act 2: dodge commercial pressure.
- Act 3: rhythm through body text.
- Boss: survive footer/popup/noise collapse.
- Outro: score curve and ghost comparison.

This turns lenses into little dramas rather than flat parameter mappings.

## 16.6 The editor: Style and Sound Lab

Creators tune sensory translation:

- Palette rules.
- Texture rules.
- Motion rules.
- Synthetic/human texture handling.
- Tempo mapping.
- Hit sounds.
- Ambient layers.
- Combo motifs.
- Reduced-motion variants.
- High-contrast variants.

Creator lenses should be known not only by mechanics, but by feel.

## 16.7 The editor: Scoring Lab

Creators should be able to compose scoring contracts from approved components:

- Accuracy.
- Efficiency.
- Endurance.
- Combo.
- Discovery.
- Risk.
- Recovery.
- Route quality.
- Sort correctness.
- Timing precision.

But they should not be able to create arbitrary scoring code. Score contracts must be validated and namespaced.

## 16.8 The editor: Validator

The validator is non-negotiable. It checks:

- Solvability.
- Frame rate.
- Mobile playability.
- Readability.
- Contrast.
- Reduced-motion compliance.
- Degenerate scoring.
- Bot exploitability.
- Failure on test packs.
- Excessive volatility.
- Unsafe content reproduction.
- Misleading page claims.
- Leaderboard integrity.

A lens that fails can be played privately but cannot be publicly listed.

## 16.9 AI co-designer

AI should assist creators by translating ideas into draft graphs. A creator might type:

"Make recipe pages feel like a heist where the player steals the actual recipe from a maze of ads and personal anecdotes."

The AI proposes:

- Best page type: recipe.
- Verbs: COLLECT + SURGE + DEFEND.
- Signals: ingredients, instruction headings, ad zones, jump links, newsletter overlays.
- Acts: preamble fog, ad guards, ingredient vault, instruction escape.
- Scoring: recipe extraction efficiency, ad avoidance, combo.

The creator edits the graph, tests it, validates it, and publishes. The AI suggests; the platform constrains.

## 16.10 No arbitrary code

Public lenses should not run arbitrary JavaScript. They should be declarative graphs interpreted by WebWave's safe runtime.

Layers:

- Beginner: prompts, presets, sliders.
- Intermediate: node graph.
- Advanced: safe declarative DSL.
- Internal/pro: engine scripting for trusted official work.

This is essential for security, moderation, performance, anti-cheat, and platform review.

## 16.11 Publication tiers

Lens publishing should move through trust tiers:

Draft: creator only.

Private: creator and selected testers.

Unlisted Share: anyone with link.

Community Listed: visible but not featured.

Certified: passed broader test packs.

Featured: selected by WebWave editorial.

Canon Candidate: being evaluated for official integration.

Officialized: hardened and added to the official grammar.

This prevents the Lens Market from becoming unbounded sludge.

## 16.12 Marketplace of lenses and pairings

The market should show both lenses and URL/lens pairings.

Lens example:

Ad Gauntlet

- Best on: news, recipes, affiliate sites.
- Verbs: SURGE + DEFEND.
- Feels like: fighting attention extraction.
- Fun-yield: 74% on certified test pack.
- Accessibility: medium.

Pairing example:

- URL: a specific recipe page.
- Lens: Recipe Heist.
- Why featured: unusually clean ingredient/instruction extraction and hilarious ad pressure.

The product should celebrate both creators and cartographers.

## 16.13 Example mature lenses

Ad Gauntlet: CTAs, ads, sticky headers, and popups become waves of attention pressure.

Recipe Heist: ingredients and instructions are the treasure; ads and preamble are guards and fog.

Corporate Soulslike: SaaS pages become hostile architecture, with abstract buzzwords as fog and CTAs as gates.

Community lenses should not fragment the official game. Every URL still has an Official Run. Lenses are alternate interpretations with separate namespaces.

## 18.5 Performance targets

Mature targets:

- Extension idle state: no meaningful CPU, no network.
- Extraction after invocation: fast enough to feel intentional, ideally under 500ms for common pages.
- Cached load: near-instant, ideally under 200ms from invocation to game shell.
- Novel page load: ideally under 2 seconds for most eligible pages, with graceful "reading this page" animation.
- Gameplay: stable 60fps where possible.
- Offline buffering: scores and traces submit later.
- Reduced-motion and relaxed modes maintain clarity.

## 18.6 Browser extension constraints

Manifest V3 service workers are event-driven and can terminate. The architecture should not rely on persistent global state in the extension worker. Important state should live in storage, server records, or the active game runtime.

Temporary active-tab style invocation should be favored where possible for trust and permission minimization.

# 22. Competitive Integrity

## 22.1 Scoring philosophy

Scoring should be simple enough to understand and deep enough to master. Each run produces one leaderboard score, but the player can inspect dimensions.

Common dimensions:

- Accuracy.
- Efficiency.
- Endurance.
- Combo mastery.
- Risk/reward.
- Route quality.
- Recovery.

## 22.2 Normalized scores

Cross-page comparison needs normalization. Raw scores are local. Normalized scores compare performance against page distributions.

Use distributions carefully. Early pages rely on AI benchmarks and provisional estimates. Mature pages use human median, variance, percentiles, and score-shape data.

## 22.3 Anti-cheat

Mature anti-cheat measures:

- Server-side score validation.
- Replay/trace consistency checks.
- Input timing anomaly detection.
- Bot benchmark comparisons.
- Rate limits.
- Device/viewport class separation.
- Suspicious perfect-score flags.
- Lens-specific exploit detection.
- Public leaderboard eligibility rules.

Self-hosted or low-complexity pages should default to private or friend leaderboards until verified.

# 23. Risks and Mitigations

## 23.1 Risk: core templates are not fun

Severity: existential.

Mitigation: every verb must be a good game before it is a web interpreter. Playtest official verbs in isolation. Kill or redesign weak verbs. Do not hide weak game feel under AI novelty.

## 23.2 Risk: page-to-game connection feels fake

Severity: high.

Mitigation: test whether players can match a played WebWave level to one of several source pages. Improve sensory translation, section-based acts, semantic mapping, and page portraits until players feel the connection.

## 23.3 Risk: templates feel samey

Severity: high.

Mitigation: multi-act compilation, hybrid verbs, strong sensory layer, page anatomy, better curation, creator lenses, and fun-yield analytics by page class.

## 23.4 Risk: privacy backlash

Severity: high.

Mitigation: summon model, active-tab-style access where possible, local extraction, no raw content by default, visible extraction report, open-source extractor, sensitive blocklists, clear policy, domain/user controls.

## 23.5 Risk: unsafe or offensive pages become public game objects

Severity: high.

Mitigation: public/private/block tiers, safety classifier, human review for flagged/featured pages, community reporting, no public leaderboard until eligibility clears, strict creator moderation.

## 23.6 Risk: legal challenges from domains

Severity: medium to high.

Mitigation: abstract feature extraction, no screenshots/content reproduction, robots.txt respect for prewarming, domain claims and opt-outs, legal counsel, careful sponsorship and archive policy.

## 23.7 Risk: creator market floods product with low-quality modes

Severity: medium to high.

Mitigation: canonical-first hierarchy, publication tiers, certification, editorial featuring, validation, separate leaderboards, quality-weighted discovery, no arbitrary code.

## 23.8 Risk: monetization misaligns incentives

Severity: medium.

Mitigation: no pay-to-win, sponsored page honesty, creator quality metrics, premium as depth and convenience, domain analytics only aggregate.

## 23.9 Risk: technical complexity overwhelms team

Severity: high.

Mitigation: build maturity architecture through internal tools first. Prove one verb and compiler path. Keep AI bounded. Delay Lens Market until official system is strong.

## 23.10 Risk: daily ritual fails to retain

Severity: high.

Mitigation: improve page curation, share-card quality, ghost comparison, daily streak, seasonal trails, and mobile load. The daily run must be consistently good.

# 28. Glossary

Atlas: The browsable map of played URLs, domains, histories, lenses, and discoveries.

Canonical Run: The official WebWave interpretation of a page and version.

Compiler: The system that turns page signals into level scripts.

Feature Signature: Abstract representation of page structure, rhythm, media, density, aesthetic, and intent.

Fun-yield: Percentage of pages in a class that produce enjoyable, fitting, fair gameplay.

Ghost: A trace of a prior run used for asynchronous social presence.

Lens: A creator-authored or official interpretation system that maps page signals into playable verbs and acts.

Level Script: The compiled output containing acts, events, scoring, visuals, audio, and validation.

Page Portrait: A stylized visual signature generated from page features, not a screenshot.

Page Version: A meaningful playable state of a URL.

Playable Link: The core share object: a URL, version, mode/lens, score, ghost, and direct play entry.

Summon Model: Extension interaction pattern where processing begins only after explicit user invocation.

Synthetic Regularity: Internal signal for smoothness, repetition, homogeneity, or AI-like texture, not a public authorship claim.

Verb: A core gameplay action family such as SURGE, COLLECT, RHYTHM, BALANCE, TRACE, DEFEND, or STACK.
